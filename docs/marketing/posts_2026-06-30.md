## Why I'm posting here

This sub hates telemetry, accounts, and cloud-only apps. I built a habit tracker that respects all three:
- 100% local, Windows desktop app
- ~3 MB installer, no Electron, no Chromium-bundled-with-your-app
- No account, no analytics, no update server
- Data lives in a single localStorage entry with FNV-1a checksum + backup key

But the reason I'm posting today is that **I ran a 4-agent audit on the codebase before tagging v0.2.1, and it found 9 real bugs** — some of them HIGH severity. I think the process is worth sharing because most side projects never get audited at all.

## The 9 bugs (all fixed, all tested)

### HIGH
1. **Save race condition** — calling save() twice in quick succession (e.g. checking off two habits fast) would drop the second write if the first hadn't finished yet. Fixed with a pendingData slot.
2. **Undo/redo resurrects ghost data** — restoring a snapshot from before a habit was deleted would recreate dangling check-ins for the dead habit. Now guarded.
3. **Import collision** — re-importing a backup that contained duplicate habit IDs (some export formats allow this) would silently attach the data to the wrong habit. Fixed with a seenImportIds Set.
4. **Auto-restore broken in dev** — a module-level flag meant React StrictMode's double-mount in dev would set "already checked" on the first mount, so the second mount skipped restore entirely. Moved to useRef.

### MEDIUM
5. **Safety net misses note-only changes** — the 100-entry autosave recovery trigger only counted check-ins. Saving a habit note alone wouldn't fire. Now includes `notes.length > 0`.
6. **Chaos view rescans every render** — `useMemo([tick])` wrap on the report computation.
7. **Date validator accepted `2026-02-30`** — JS `new Date("2026-02-30")` silently rolls over to March 2. Now regex + round-trip validated.

### LOW
8. **Silent backup failure** — backup write errors were swallowed. Now `console.warn`.
9. **(Lighthouse UX)** — empty habits show a flat-line sparkline. Kept as-is for now.

## Process notes

- 4 parallel sub-agents audited the codebase, each with a different lens (data integrity, React lifecycle, persistence, TS strictness)
- Every fix has a regression test
- 231 tests passing (was 223, +8)
- TypeScript: 0 errors. ESLint: 0 errors, 0 warnings.

If you're building local-first apps, **the auto-restore bug (#4) is the one I'd look for in your own code** — module-level mutable state + React's StrictMode is a known footgun that most dev environments hide because dev remount ≠ prod remount.

## Stack

- Tauri v2 (Rust + WebView2)
- React 19 + TypeScript (strict)
- Vite
- localStorage + FNV-1a checksum + backup key
- Pure CSS, no UI lib
- 231 tests (Vitest + RTL)

## Install

- GitHub: https://github.com/Lemniscate-world/LifeTrack
- Release v0.2.1: https://github.com/Lemniscate-world/LifeTrack/releases/tag/v0.2.1
- MSI (2.91 MB) and NSIS (1.94 MB) installers available

## TL;DR

A 4-parallel-agent code review on a ~3,000 LOC Tauri + React app surfaced 9 real bugs. Here are the 3 patterns I now check first in any local-first app.

## Pattern 1: Save race in write-through-cache

Symptom: user clicks two buttons fast, second write is lost.
Cause: `save(data)` returns a Promise; calling it twice while the first is still resolving can drop the second payload.

```ts
// BAD: second call can be dropped
async function save(data) {
  if (inFlight) return; // <-- this!
  inFlight = true;
  await persist(data);
  inFlight = false;
}
```

Fix: keep a `pendingData` slot. If `save()` is called while `inFlight`, stash the latest data in `pendingData` and replay it after the current save resolves.

## Pattern 2: Undo/redo with deleted entities

Symptom: undoing an edit made before a habit was deleted resurrects the habit's data.
Cause: undo restores the entire snapshot without checking if referenced entities still exist.

Fix: before applying the snapshot, filter restored data against `data.habits.some(h => h.id === entry.habitId)`.

## Pattern 3: Module-level mutable state + React StrictMode

Symptom: a one-shot init only fires half the time in dev.
Cause: `let didInit = false; if (didInit) return; didInit = true;` — in dev, StrictMode mounts twice, so the second mount sees `didInit === true` and skips.

Fix: use `useRef` inside the component, not module state.

## Bonus pattern: silent date validation

```ts
new Date("2026-02-30") // → "2026-03-02" without complaint
```

The Date constructor will happily roll over invalid days. Always round-trip:
```ts
function isValidISODate(s: string) {
  if (!/^\d{4}-\d{2}-\d{2}$/.test(s)) return false;
  const d = new Date(s + "T00:00:00Z");
  return d.toISOString().slice(0, 10) === s;
}
```
